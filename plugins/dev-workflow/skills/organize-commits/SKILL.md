# コミット整理スキル

ブランチ上のコミット履歴を分析し、論理的なグループに再編成する。
レビュー後に散らかったコミット（fixコミット混在、無関係な変更など）を整理するためのスキル。

## 前提条件

- 整理対象のブランチにチェックアウトしていること
- ワーキングツリーがクリーンであること（未コミットの変更がないこと）

## Phase 1: 分析（読み取り専用）

### 1. ワーキングツリーのクリーン確認

```bash
git status --porcelain
```

出力がある場合は「未コミットの変更があります。先にコミットまたはスタッシュしてください。」と表示して終了する。

### 2. ベースブランチの決定

- `{{base-branch}}` が指定されている場合はそれを使用する
- 指定がない場合は以下の順で自動検出する:
  1. `git rev-parse --abbrev-ref origin/HEAD` を試す（`origin/main` などから `main` を取得）
  2. 失敗した場合は `main` → `master` の順でブランチの存在を確認
  3. いずれも見つからない場合はユーザーにベースブランチ名の入力を求める

現在のブランチ名を取得する:

```bash
git rev-parse --abbrev-ref HEAD
```

### 3. コミット一覧の取得

```bash
git log --oneline <base>..HEAD
```

コミットが0件の場合は「ベースブランチとの差分がありません。」と表示して終了する。
各コミットのSHAとサブジェクトを記録する。

### 4. PRコンテキストの取得

PRの目的を把握するため、以下を試みる:

```bash
gh pr view --json title,body,labels 2>/dev/null
```

PRが存在しない場合はコミットメッセージ全体からブランチの目的を推測する。

### 5. 各コミットの変更内容の調査

コミット数が多い場合（10件超）は処理に時間がかかることをユーザーに伝える。
各コミットのSHAに対して以下を実行し、変更内容を記録する:

```bash
git show --stat <sha>
git show <sha>
```

各コミットについて以下を記録する:
- 変更されたファイル一覧
- 変更の種類（ロジック変更、テスト追加、設定変更、フォーマット変更など）
- コミットメッセージ

### 6. 全体の変更セットの確認

```bash
git diff <base>...HEAD --stat
git diff <base>...HEAD
```

再編成の前後検証に使う差分ハッシュをメモしておく:

```bash
git diff <base>...HEAD | git hash-object --stdin
```

### 7. クロスリファレンス分析

PRの目的・コミットメッセージと実際の変更内容を照合し、以下を検出する:

**無関係な変更**（PRの目的と関連しないもの）:
- 別機能のリファクタリング
- 無関係なファイルのフォーマット修正
- PRスコープ外の設定変更

**デバッグ・一時コード**:
- `console.log`, `console.debug`, `puts`, `binding.pry`, `binding.irb`, `debugger` の残存
- `TODO: remove`, `FIXME`, `HACK` コメント
- テスト用のハードコードされた値

**フォーマットのみの変更がロジック変更と混在**:
- インデント修正や空白変更がロジック変更と同じコミットに含まれている

**重複する変更**:
- 同じファイルを複数コミットで変更→戻す→再変更しているパターン

### 8. グルーピング案の提示

以下の形式でユーザーに分析結果と提案を表示する:

```
## 現在のコミット構成（<N>件）

1. abc1234 feat: ユーザー認証機能の追加
2. def5678 fix: レビュー対応 - 変数名の修正
3. ghi9012 test: 認証テストの追加
4. jkl3456 fix: レビュー対応 - エラーハンドリング追加
5. mno7890 fix: レビュー対応 - テスト修正

## 提案するコミット構成

### コミット 1: feat: ユーザー認証機能の追加
- app/models/user.rb（ロジック変更）
- app/controllers/sessions_controller.rb（新規）
- config/routes.rb（ルーティング追加）
（元コミット: 1, 2, 4 をマージ）

### コミット 2: test: 認証機能のテスト追加
- spec/models/user_spec.rb（新規）
- spec/controllers/sessions_controller_spec.rb（修正）
（元コミット: 3, 5 をマージ）

## ⚠️ 要確認: 無関係と思われる変更

- app/helpers/application_helper.rb: インデント修正のみ（PRの目的と無関係）
  → 含める / 別コミットにする / 除外して破棄する

## ⚠️ 要確認: デバッグコードの残存

- app/models/user.rb:42: `puts "debug: #{token}"` が残っています
  → 削除する / 残す
```

### 9. ユーザーの確認を得る

以下について明示的な承認を求める:
- グルーピング案の承認または修正指示
- 無関係な変更の扱い（含める / 別コミット / 破棄）
- デバッグコードの扱い（削除 / 残す）

**ユーザーの明示的な承認を得るまで、Phase 2には絶対に進まないこと。**

---

## Phase 2: 再編成（破壊的操作）

Phase 1でユーザーの承認を得た後のみ実行する。

### 1. バックアップブランチの作成

```bash
git branch backup/<current-branch>-$(date +%Y%m%d-%H%M%S)
```

作成されたバックアップブランチ名をユーザーに表示する。

### 2. 変更内容のハッシュ保存

再編成前の差分ハッシュを記録する（Phase 1 ステップ6で取得した値を使用）。

### 3. mixedリセットの実行

全変更をunstaged状態でワーキングツリーに戻す:

```bash
git reset <base>
```

注意: `--soft` ではなく `--mixed`（デフォルト）を使う。
staged状態だと選択的な `git add` ができないため。

### 4. 承認されたグルーピングに従いコミット

各グループを順番に処理する:

1. 対象ファイルをステージング:
   ```bash
   git add <file1> <file2> ...
   ```
2. ステージング内容を確認:
   ```bash
   git diff --cached --stat
   ```
3. コミット:
   ```bash
   git commit -m "<承認されたコミットメッセージ>"
   ```

**ファイル内の一部変更のみを別コミットに分ける必要がある場合**:
`git add -p` は対話的操作のためClaudeが自動実行できない。
ユーザーに「`git add -p <filename>` を手動で実行してください」と案内し、
完了したら継続を伝えてもらう。

**除外が承認された変更がある場合**:
全コミット完了後、除外対象のファイルをユーザーに最終確認してから破棄する:
```bash
git checkout -- <file>
```

### 5. 差分ハッシュの検証

再編成後の差分ハッシュを計算し、Phase 1 ステップ6の値と比較する:

```bash
git diff <base>...HEAD | git hash-object --stdin
```

- **一致**: 再編成成功。変更内容に差異なし。
- **不一致かつ意図的な除外がない場合**: エラーとして以下を表示して終了する:
  ```
  ❌ 差分ハッシュが一致しません。変更が失われた可能性があります。
  バックアップから復旧するには:
    git reset --hard backup/<branch>-<timestamp>
  ```
- **不一致かつ意図的な除外がある場合**: 除外した変更分の差異であることをユーザーに確認する。

### 6. 完了メッセージ

以下の情報をユーザーに表示する:

```
✅ コミット整理が完了しました

## 再編成後のコミット構成
（git log --oneline <base>..HEAD の出力を表示）

## バックアップ
- バックアップブランチ: backup/<branch>-<timestamp>
- 復旧方法: git reset --hard backup/<branch>-<timestamp>
- バックアップが不要になったら: git branch -d backup/<branch>-<timestamp>

## リモートへのプッシュ
リモートに既にプッシュ済みの場合、履歴が書き換わるため強制プッシュが必要です:
  git push --force-with-lease origin <branch>
（--force ではなく --force-with-lease を使うと、他者が push していた場合に安全に失敗します）
```

## 注意事項

- Phase 2は破壊的操作のため、必ずユーザーの明示的な承認を得てから実行する
- バックアップブランチは必ず作成する（省略不可）
- 差分ハッシュの検証は全コミット完了後に1回行う
- リモートへの強制プッシュはこのスキルでは実行しない（手順を案内するのみ）
- `git add -p` が必要なケースはユーザーに手動操作を依頼する
